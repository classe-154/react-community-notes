## 🌐 1. **Scope e Hoisting**

Man mano che i programmi diventano più grandi, diventa fondamentale capire dove una variabile è visibile e può essere usata, e dove invece è invisibile. Questo concetto si chiama Scope (Ambito di visibilità).

Inoltre, Javascript ha un comportamento particolare chiamato Hoisting, che consiste nel muovere le dichiarazioni di funzioni e variabili all'inizio del programma prima di avviare l'esecuzione.

Capire lo Scope e l'Hoisting ti eviterà i bug più frustranti della programmazione, quelli in cui il computer si blocca dicendo che una variabile "non esiste", anche se tu sei sicuro di averla scritta poche righe più in là.

## 🔍 2. **Cos'è lo Scope (E la differenza tra Let, Const e Var)**

Lo Scope è il raggio d'azione di una variabile, ovvero la zona del programma in cui quella variabile è "viva" e accessibile. In JavaScript esistono due ambiti principali:

- **Scope Globale:** Le variabili create fuori da qualsiasi parentesi graffa `{}` sono globali. Sono visibili ovunque, in qualsiasi punto del file.
    
- **Scope di Blocco:** Le variabili create dentro un blocco di parentesi graffe `{}` (come un if o un ciclo) vivono solo lì dentro. All'esterno di quelle graffe, smettono di esistere.

- **Scope di Funzione (Locale):** Le variabili create dentro le `function` vivono ed esistono solo all'interno di quella funzione, e vengono create solo quando la funzione viene invocata.

Qui entra in gioco il motivo per cui `let` e `const` hanno sostituito la vecchia parola chiave `var`:

- `let` e `const` rispettano lo Scope di Blocco. Se le crei dentro un if, fuori da quell'if nessuno potrà toccarle. Sono sicure.
    
- `var` **NON** rispetta lo Scope di Blocco. Se crei una variabile con `var` dentro un if, quella variabile "scappa" fuori dalle parentesi graffe e diventa accessibile in tutto il file, rischiando di sovrascrivere o sporcare altri dati per errore.
    

```js
// SCOPE GLOBALE
let messaggioGlobale = "Sono visibile ovunque!";

if (true) {
  // SCOPE DI BLOCCO
  let messaggioLocale = "Vivo solo dentro questo if";
  var messaggioVarScappato = "Sono un var, me ne frego delle graffe!";
  
  console.log(messaggioGlobale); // Funziona!
  console.log(messaggioLocale);  // Funziona!
}

console.log(messaggioVarScappato); // Funziona! (Var è scappata)
console.log(messaggioLocale);      // ERRORE! messaggioLocale non esiste qui fuori
```

### 🛗 3. **Cos'è l'Hoisting**

La parola Hoisting significa "innalzamento". È un meccanismo automatico in cui JavaScript, prima di eseguire il codice, legge tutto il file e "sposta" virtualmente in cima la dichiarazione delle variabili e delle funzioni.

Tuttavia, il modo in cui viene gestito cambia totalmente in base a come abbiamo creato la variabile:

- **Con var:** JavaScript sposta la variabile in cima ma non il suo valore. La variabile viene inizializzata come `undefined`. Se provi a usarla prima della sua dichiarazione, il programma non si rompe, ma ti restituisce un valore vuoto.
    
- **Con let e const:** JavaScript blocca questo comportamento selvaggio. Le variabili vengono sì rilevate e spostate in cima, ma inserite in una zona di sicurezza chiamata **Temporal Dead Zone** (Zona Morta Temporale). Se provi a usare una variabile `let` o `const` prima di averla dichiarata, il programma si blocca immediatamente con un errore di protezione (`ReferenceError`).
    

```js
// Tentativo di utilizzo prima della dichiarazione

console.log(variabileVar); // Restituisce 'undefined' (Non si rompe, ma è vuota)
var variabileVar = "Contenuto Var";

console.log(variabileLet); // ERRORE IMMEDIATO! Il programma si blocca qui
let variabileLet = "Contenuto Let";
```

### ⚠️ 4. **Errori comuni**

- **Dichiarare una variabile dentro l'if e pretendere di usarla fuori:** Questo è l'errore tipico quando si calcolano i prezzi. Se scrivi `if (sconto) { let prezzoFinale = 10; }` e poi provi a fare il `console.log(prezzoFinale)` fuori dalle graffe, il programma fallirà. Devi dichiarare la variabile prima dell'if con `let prezzoFinale;` (senza valore) e poi valorizzarla dentro le graffe.
    
- **Affidarsi all'hoisting di var scrivendo codice disordinato:** Usare le variabili prima di averle create rende il codice difficilissimo da leggere e tracciare. Anche se con `var` il programma non si rompe e restituisce `undefined`, è una pessima pratica. Usa sempre `let` e `const` per obbligarti a seguire un ordine logico e pulito.
    

## 📌 5. **Approfondimento leggero: La catena dello Scope (Scope Chain)**

JavaScript ragiona sempre dall'interno verso l'esterno. Se ti trovi dentro un blocco di parentesi graffe molto profondo e chiedi una variabile, JavaScript:

1. Cerca prima dentro il blocco corrente.
    
2. Se non la trova, fa un passo indietro e guarda nel blocco più esterno (il "genitore").
    
3. Continua a risalire fino ad arrivare allo Scope Globale.
    

Se non la trova nemmeno lì, si arrende e lancia un errore. Il percorso inverso, invece, è impossibile: un blocco esterno non può mai guardare cosa succede dentro le parentesi graffe dei suoi figli.

## 7. **Key Takeaways del Giorno**

- **Le graffe isolano (let/const):** Ogni volta che vedi delle parentesi graffe `{ }`, sappi che le variabili create all'interno con `let` o `const` nascono e muoiono esclusivamente lì dentro.
    
- **Addio a Var:** L'uso di `var` è sconsigliato nella programmazione moderna perché ignora lo scope di blocco e genera comportamenti imprevedibili a causa dell'hoisting libero.
    
- **Dichiarare prima di usare:** Con `let` e `const` sei protetto: JavaScript ti impedirà tassativamente di utilizzare una variabile prima che tu l'abbia effettivamente creata.
    

## 8. **Glossario**

| **Termine Istituzionale** | **Definizione Formale**                                                                             | **"Spiega Brutta"**                                                                                                    |
| ------------------------- | --------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Scope**                 | L'ambito di visibilità e il ciclo di vita di una variabile all'interno del codice.                  | Il territorio in cui una variabile è viva e può essere chiamata per nome senza far arrabbiare il computer.             |
| **Block Scope**           | Proprietà delle variabili limitate al blocco di istruzioni racchiuso tra graffe `{ }`.              | La regola del "quello che succede nelle graffe, resta nelle graffe" (valida per let e const).                          |
| **Hoisting**              | Comportamento del motore JavaScript che solleva le dichiarazioni in cima allo scope corrente.       | L'abitudine di JavaScript di leggere in anticipo i nomi delle variabili prima di far partire il programma.             |
| **Temporal Dead Zone**    | Periodo che va dall'inizio dello scope alla dichiarazione effettiva di una variabile `let`/`const`. | La zona d'ombra in cui una variabile esiste già per il computer ma è illegale toccarla finché non ci arrivi a leggere. |
