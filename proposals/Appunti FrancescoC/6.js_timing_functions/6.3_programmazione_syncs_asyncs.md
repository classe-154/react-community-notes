## 🔄 6.3 Programmazione Sincrona vs Asincrona (L'Event Loop)

Di base, JavaScript è un linguaggio Sincrono e Single-Threaded. Significa che possiede un solo "motore" per fare i calcoli e può eseguire una sola riga di codice alla volta, seguendo un ordine rigido dall'alto verso il basso. Se la riga 2 richiede molto tempo per essere elaborata (ad esempio un calcolo matematico gigantesco), la riga 3 rimane bloccata in attesa (comportamento bloccante).

Tuttavia, grazie al browser, JavaScript può gestire operazioni Asincrone. L'asincronia permette di avviare un'operazione a lungo termine (come un timer) e di "metterla in sottofondo", continuando immediatamente a eseguire il resto del codice senza bloccare la pagina.

### ⏳ 6.3.1 Il paradosso del timer a zero millisecondi

Per capire come JavaScript gestisce il tempo e la precedenza del codice, analizziamo questo famosissimo esempio. Cosa viene stampato prima in console?

```JavaScript
console.log("1. Inizio Script");

// Impostiamo un timer a ZERO millisecondi!
setTimeout(() => {
  console.log("2. Dentro il Timer");
}, 0);

console.log("3. Fine Script");
```

Logicamente potresti pensare: "Il timer è a 0 secondi, quindi è istantaneo, verrà eseguito nel mezzo!".

Invece l'output della console sarà:

```Plaintext
1. Inizio Script
2. Fine Script
3. Dentro il Timer
```

Perché succede questo? Perché interviene un meccanismo del browser chiamato **Event Loop**.

### ☕ 6.3.2 Come funziona l'Event Loop (La metafora del Bancone del Bar)

Per capire l'Event Loop senza impazzire sulle definizioni tecniche, immagina un bar con un solo barista (il Call Stack di JavaScript) e una fila di clienti da servire.

- **Il Call Stack (Il Barista):** Esegue il codice sincrono riga per riga. Legge `console.log("1. Inizio Script")` e lo serve subito.
    
- **Le Web APIs (La Cucina):** Quando il barista incontra il `setTimeout`, non si mette lui a contare i millisecondi. Prende l'ordine del timer e lo passa alla cucina (il browser). Il barista è subito libero di servire il cliente successivo, stampando `console.log("3. Fine Script")`.
    
- **La Callback Queue (La Cassa dei piatti pronti):** Anche se la cucina ci mette 0 secondi a preparare il piatto (il timer scade subito), il piatto pronto non viene lanciato in testa al barista. Viene messo in una fila d'attesa (la coda delle Callback).
    
- **L'Event Loop (Il Direttore di sala):** È un meccanismo che controlla continuamente il barista. Il suo unico compito è: aspettare che il barista abbia finito di servire TUTTI i clienti della fila sincrona. Solo quando il Call Stack è completamente vuoto, l'Event Loop prende il primo piatto pronto dalla coda (il codice del `setTimeout`) e lo consegna al barista per farlo eseguire.
    

Ecco perché il timer a 0 millisecondi uscirà sempre e comunque dopo l'ultimo `console.log` sincrono della pagina.

### 🗺️ 6.3.3 La Mappa del Percorso Asincrono

Per riassumere, ecco il viaggio che fa il tuo codice quando usi un `setTimeout`:

1. **Call Stack:** Il codice legge il `setTimeout` e lo lancia via per non perdere tempo.
    
2. **Web APIs (Browser):** Il browser fa partire il conto alla rovescia in sottofondo.
    
3. **Callback Queue:** Il timer scade e la funzione si mette in fila nella sala d'attesa.
    
4. **Event Loop:** Guarda il Call Stack. È vuoto? Se sì, prende la funzione dalla sala d'attesa e la ributta nel Call Stack per farla eseguire.
    

### ⚠️ 6.3.4 Errori comuni

- **Pensare che l'asincronia esegua le cose contemporaneamente:** JavaScript non fa due cose insieme. Il `setTimeout` viene delegato al browser per il conteggio del tempo, ma quando la callback torna indietro, deve comunque aspettare il suo turno in fila. Se il tuo codice sincrono principale è bloccato in un ciclo infinito, il timer non verrà eseguito mai, anche se il suo tempo è scaduto da ore.
    
- **Sottovalutare l'ordine di esecuzione:** Scrivere logiche che si aspettano di ricevere i dati di un timer nella riga subito successiva all'avvio del timer stesso genererà bug e valori `undefined`. Tutto ciò che è asincrono va gestito esclusivamente dentro la funzione di callback.

🚀 **Key Takeaways del Giorno**

- **Il codice sincrono ha sempre la precedenza:** JavaScript svuota completamente la sua pila di esecuzione principale prima di andare a guardare se ci sono timer pronti.
    
- **L'asincronia non è magia:** È un sistema di gestione delle code. Il browser sposta i compiti lenti in sottofondo per evitare di congelare l'interfaccia grafica del sito.
    
- **Il timer a 0 è un rinvio:** Usare `setTimeout` con 0 millisecondi serve tecnicamente a dire a JavaScript: "Esegui questa azione il prima possibile, ma solo dopo aver finito tutto il resto".
    

📖 **Glossario**

| **Termine Istituzionale** | **Definizione Formale**                                                                                   | **"Spiega Brutta"**                                                                                                           |
| ------------------------- | --------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **Call Stack**            | Struttura dati che tiene traccia del punto in cui si trova l'esecuzione del programma JavaScript.         | Il barista: la pila di comandi sincroni che JavaScript sta eseguendo riga per riga in questo momento.                         |
| **Callback Queue**        | Coda in cui vengono accumulati i messaggi e le funzioni asincrone pronte per essere elaborate.            | Il bancone dei piatti pronti: la fila dove si mettono le callback dei timer scaduti in attesa che il barista si liberi.       |
| **Event Loop**            | Ciclo di controllo che sposta le funzioni dalla Callback Queue al Call Stack quando quest'ultimo è vuoto. | Il direttore di sala: il meccanismo che controlla se il barista ha finito i compiti e gli passa i piatti pronti dalla cucina. |
