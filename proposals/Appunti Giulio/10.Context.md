# 🌀 Guida Definitiva a React Context: Gestione dello Stato Globale, Custom Hooks e Carrello Elettronico


## 🕳️ 1. Il Problema: Prop Drilling (Caso Studio: Switch Tema)

Nelle applicazioni React standard, i dati si muovono dall'alto verso il basso (flusso di dati unidirezionale). Quando un componente profondo ha bisogno di un'informazione gestita in un componente antenato, si genera il fenomeno del **Prop Drilling**.

### 🚨 L'Anatomia del Problema

* **Il Genitore (`App.jsx`):** Detiene la "fonte della verità" (lo stato `theme`).


* **L'Intermediario (`Header.jsx`):** Funge esclusivamente da "postino". Riceve le props `{ theme, setTheme }` e le passa al figlio, senza utilizzarle per la propria logica o stile.


* **Il Target (`ThemeButton.jsx`):** Consuma finalmente il dato per scattare lo switch del CSS.



### ⚠️ I Segnali di Allarme

1. **Fragilità:** Se rinominiamo una prop nel genitore, dobbiamo aggiornare manualmente tutta la catena di componenti intermedi.


2. **Inquinamento del codice:** I componenti intermedi perdono la loro indipendenza e diventano difficili da riutilizzare altrove perché "ingombrati" da props estranee.


3. **Efficienza:** React esegue il re-render automaticamente quando una props o uno stato cambiano per un componente. Quindi tutti quei componenti intermedi il cui unico scopo è di passare le props ai loro figli, vengono re-renderizzati quando cambia quella prop anche se noi vogliamo solo far re-renderizzare il figlio.

## 🌀 2. La Soluzione: React Context API (Creazione & Distribuzione)

La Context API elimina la figura del "postino", creando un **canale preferenziale globale** dove i dati saltano la gerarchia dei componenti per arrivare direttamente a destinazione.

### 🏗️ Architettura in Due Fasi: `ThemeContext.jsx`

Il contesto richiede la separazione concettuale tra il **Canale** (il punto di ascolto) e il **Provider** (la sorgente di dati). Spostando lo stato nel Provider, liberiamo completamente `App.jsx` da compiti di gestione.

```jsx
import { useState, createContext } from "react";

// 1. Creazione del canale di comunicazione (valore iniziale null per sicurezza)
export const ThemeContext = createContext(null);

// 2. Creazione della Sorgente (Provider)
export function ThemeProvider({ children }) {
    // Lo stato globale vive qui dentro 🧠
    const [theme, setTheme] = useState('light');

    // Impacchettiamo lo stato e la sua funzione di aggiornamento
    const value = {
        theme,
        setTheme
    };

    return (
        // Distribuiamo l'oggettone tramite la prop obbligatoria "value"
        <ThemeContext value={value}>
            {children} {/* Permette il nesting di qualsiasi componente */}
        </ThemeContext>
    );
}

```

### 🧹 Pulizia Totale nell'Applicazione

Grazie al Provider, i componenti intermedi riacquistano la loro leggerezza nativa.

* **`App.jsx` (La Centrale):** Si limita ad avvolgere l'albero dei componenti.


```jsx
import { ThemeProvider } from "./contexts/ThemeContext";
import Header from "./components/Header";
import Main from "./components/Main";

function App() {
    return (
        <ThemeProvider>
            <div className="container">
                <Header /> {/* Pulito! Nessuna prop passata 🧹 */}
                <Main />
            </div>
        </ThemeProvider>
    );
}

```


* **`Header.jsx` (L'ex Postino):** Non tocca più le props del tema, si limita a renderizzare il bottone.

## 🎣 3. L'Evoluzione Professionale: Custom Hooks & Sicurezza (`useTheme`)

Invece di costringere ogni componente a importare sia `useContext` sia il file specifico `ThemeContext`, lo standard professionale prevede l'astrazione di questa logica in un **Custom Hook** protetto.

### 🛠️ Implementazione del Connettore: `src/hooks/useTheme.js`

```javascript
import { useContext } from 'react';
import { ThemeContext } from '../contexts/ThemeContext';

function useTheme() {
    const themeContext = useContext(ThemeContext);

    // 🛡️ Guardrail di Sicurezza: evita errori silenziosi in fase di sviluppo
    if (themeContext === null) {
        throw new Error('Ti sei scordato il Provider! Verifica che il componente sia dentro ThemeProvider.');
    }

    return themeContext; // Ritorna l'oggetto { theme, toggleTheme }
}

export default useTheme;

```

### 🔄 Ottimizzazione delle Funzioni nel Provider

Per rendere i componenti UI ancora più snelli, possiamo delegare le funzioni di modifica direttamente all'interno del Provider:

```jsx
// Dentro ThemeContext.jsx
const toggleTheme = () => {
    setTheme(prev => (prev === 'light' ? 'dark' : 'light'));
};

const value = { theme, toggleTheme };

```

### 🖱️ Utilizzo Semplificato in `ThemeButton.jsx`

```jsx
import useTheme from "../hooks/useTheme";

function ThemeButton() {
    // Estrazione pulita con una sola riga di codice 🎣
    const { theme, toggleTheme } = useTheme();

    return (
        <button 
            className={`btn ${theme === 'light' ? 'btn-dark' : 'btn-light'}`} 
            onClick={toggleTheme}
        >
            Cambia tema (Attuale: {theme})
        </button>
    );
}

```

## 🛒 4. Stato Globale Avanzato: Gestione Carrello (`ProductContext`)

Quando la logica globale richiede manipolazioni complesse (come l'aggiunta condizionale di elementi nel carrello), la Context API unita ai metodi immutabili degli array mostra tutta la sua potenza.

### 🏗️ Logica Avanzata di Aggiunta: `ProductContext.jsx`

```jsx
import { useState, createContext } from "react";

export const ProductContext = createContext(null);

export function ProductProvider({ children }) {
    const [products, setProducts] = useState([]);
    const [cartProducts, setCartProducts] = useState([]); // Array di oggetti { id, qty }

    const addToCart = (productId) => {
        // 🔍 Verifica l'esistenza del prodotto nel carrello
        const productFound = cartProducts.find(product => product.id === productId);

        if (productFound === undefined) {
            // Caso A: Nuovo prodotto -> Lo aggiungiamo preservando la storia (...) e impostando qty a 1
            setCartProducts([
                ...cartProducts,
                { id: productId, qty: 1 }
            ]);
        } else {
            // Caso B: Prodotto già esistente -> Generiamo un nuovo array con .map() e incrementiamo solo la qty
            const updatedCart = cartProducts.map(product => {
                if (product.id === productId) {
                    return { ...product, qty: product.qty + 1 };
                }
                return product; // Gli altri prodotti restano invariati
            });
            setCartProducts(updatedCart);
        }
    };

    return (
        <ProductContext value={{ products, setProducts, cartProducts, addToCart }}>
            {children}
        </ProductContext>
    );
}

```

### 📝 Regole d'oro per lo Stato Immutabile del Carrello

* **`.find()`:** Essenziale per ispezionare l'array. Restituisce il riferimento all'oggetto se esiste, altrimenti `undefined`.


* **Operatore Spread (`...`):** Utilizzato per copiare i vecchi elementi nel nuovo array senza mutare lo stato originale direttamente.


* **`.map()`:** Perfetto per generare un nuovo array modificando esclusivamente l'elemento target identificato dall'ID.

## 📂 5. Architettura delle Cartelle Standard e Flusso dei Dati

Nelle applicazioni reali, dividere accuratamente le responsabilità dei file previene il caos architetturale (Principio di Singola Responsabilità).

```text
src/
├── components/         # Componenti UI atomici e riutilizzabili (Header, ThemeButton)
├── contexts/           # La "Centrale dei Dati" (Definizioni Context e Provider)
├── hooks/              # I "Connettori Sicuri" (useTheme, useProduct)
├── pages/              # Viste complete (Home, ProductSingle)
├── App.jsx             # La "Centrale Elettrica" in cui i Provider avvolgono i componenti
└── main.jsx            # Punto di montaggio sul DOM reale

```

* **`contexts/`:** Gestisce il ciclo di produzione e l'immutabilità dei dati globali.


* **`hooks/`:** Isola l'implementazione tecnologica. Se un domani passassi a un altro state manager (es. Redux o Zustand), ti basterà cambiare il codice dentro l'hook per non rompere i componenti dell'interfaccia utente.
