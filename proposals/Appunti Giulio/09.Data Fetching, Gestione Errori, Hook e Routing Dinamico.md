# 🚀 Architettura Avanzata in React: Data Fetching, Gestione Errori, Hook e Routing Dinamico

## 📡 1. Data Fetching Avanzato e Ciclo di Vita (`useEffect`)

Quando colleghiamo un'applicazione a un'API esterna, dobbiamo separare nettamente la logica di rete (Utility) dalla presentazione visiva (Componente).

### 🛠️ La Logica di Rete: `src/utils/fakeStore.js`

La funzione nativa `fetch()` gestisce le richieste asincrone tramite le *Promises*. È fondamentale ricordare che **`fetch()` non fallisce (non entra nel `.catch()`) in caso di errori HTTP come 404 o 500**; fallisce esclusivamente se la richiesta di rete non parte (es. computer offline). Dobbiamo quindi lanciare un errore manuale controllando lo stato della risposta:

```javascript
const API_URL = 'https://fakestoreapi.com/products';

export function fetchFakeStoreData() {
  return fetch(API_URL)
    .then((response) => {
      // Validazione manuale degli stati di errore HTTP
      if (response.status === 404) {
        throw new Error('Pagina non trovata'); // Interrompe il flusso e spara l'errore al .catch()
      }
      return response.json(); // Trasforma lo stream grezzo in un oggetto JS leggibile
    })
    .then((json) => json);
}

```

### 🖥️ La Gestione dello Stato nel Componente: `ProductList.jsx`

Per accogliere i dati ed evitare crash grafici, inizializziamo lo stato con lo stesso tipo di dato nativo che ci aspettiamo (in questo caso, un array vuoto `[]`).

```jsx
import { useState, useEffect } from 'react';
import { fetchFakeStoreData } from '../utils/fakeStore';

function ProductList() {
  const [products, setProducts] = useState([]);
  const [errorMsg, setErrorMsg] = useState(''); // Stato per tracciare i messaggi di errore

  useEffect(() => {
    fetchFakeStoreData()
      .then((data) => {
        setProducts(data); // Scrittura dati in caso di successo
      })
      .catch((error) => {
        // Intercettiamo l'errore lanciato dall'utility o dal crash di rete
        if (error.message === 'Pagina non trovata') {
          setErrorMsg(error.message);
        } else {
          setErrorMsg('Errore generico di caricamento');
        }
      });
  }, []); // ⚠️ Array di dipendenze vuoto: esegue la fetch SOLO al montaggio del componente. Ovvero ogni volta che il componente appare per la prima volta sullo schermo (nel caso il componente sia una pagina, questo sarebbe ogni volta che navighiamo a questa pagina da una pagina diversa)

  return (
    <div>
      {errorMsg && <h1>❌ Errore: {errorMsg}</h1>}
      <pre>{JSON.stringify(products, null, 2)}</pre>
    </div>
  );
}

export default ProductList;

```

> ♾️ **Il Rischio del Loop Infinito:** Se eseguissimo la chiamata `fetchFakeStoreData()` direttamente nel corpo del componente (fuori da `useEffect`), l'aggiornamento dello stato tramite `setProducts` causerebbe un re-render, il quale rieseguirebbe la fetch, la quale aggiornerebbe lo stato, bloccando l'applicazione in un ciclo infinito di chiamate distruttive per il server.
> 
>

## 🆔 2. Rotte Parametriche (`useParams`), Sanificazione e Regole degli Hook

Quando navighiamo su una pagina di dettaglio (es. `/products/3`), React Router ci permette di catturare la parte dinamica dell'URL tramite i parametri di rotta.

### 🧼 Sanificazione dell'Input e Sicurezza

I dati estratti dall'URL tramite `useParams()` sono **sempre stringhe**. Se il database si aspetta un ID numerico, dobbiamo effettuare una sanificazione rigorosa utilizzando `parseInt()` e verificando che l'input non sia `NaN` (Not a Number), nel caso in cui un utente digiti manualmente testo nella barra degli indirizzi.

### 🚨 La Regola d'Oro degli Hook (Infrangibile)

> **Gli Hook di React (`useState`, `useEffect`, ecc.) devono essere chiamati SEMPRE e solo al primo livello della funzione del componente.**
> 
> Non possono mai essere inseriti dentro blocchi `if`, cicli `for` o funzioni annidate. React si basa sull'ordine sequenziale fisso di chiamata degli Hook per tracciare lo stato tra i vari render. Se un `if` saltasse un Hook, l'intera catena di memoria di React verrebbe disallineata, mandando in crash l'applicazione.
> 
> 

### 📝 Implementazione Perfetta: `Product.jsx`

Per rispettare le regole, dichiariamo l'Hook a primo livello, ma gestiamo la condizione di blocco *all'interno* del suo corpo, applicando un *Early Return* (ritorno anticipato) nel JSX per bloccare il rendering difettoso:

```jsx
import { useParams } from 'react-router';
import { useState, useEffect } from 'react';
import { fetchSingleProductData } from '../utils/fakeStore';

function Product() {
  const { productId } = useParams(); // 1. Estrazione parametro stringa
  const [productData, setProductData] = useState({});
  const [errorMsg, setErrorMsg] = useState('');

  // 2. Trasformazione e sanificazione preventiva
  const productIdReal = parseInt(productId);

  // 3. HOOK AL PRIMO LIVELLO (Nessun IF all'esterno)
  useEffect(() => {
    // Il controllo condizionale si fa INTERNAMENTE all'hook
    if (isNaN(productIdReal)) {
      return; // Esce dall'effetto senza fare chiamate API inutili
    }

    fetchSingleProductData(productIdReal)
      .then((data) => setProductData(data))
      .catch((error) => setErrorMsg(error.message));
  }, [productIdReal]); // Si attiva al montaggio e se l'ID sanificato cambia

  // 4. Early Return per la presentazione visiva
  if (isNaN(productIdReal)) {
    return <h1>🕵️‍♂️ Prodotto non corretto: l'ID deve essere un numero.</h1>;
  }

  if (errorMsg) {
    return <h1>🚫 {errorMsg === 'Lunghezza 0' ? 'Prodotto non esistente' : errorMsg}</h1>;
  }

  return (
    <div>
      <h1>✨ {productData.title}</h1>
      <p>{productData.description}</p>
    </div>
  );
}

export default Product;

// Nota Bene, in questo esempio abbiamo usato una funzione fetchSingleProductData che non abbiamo definito nello snippet, ti basti sapere che è una funzione che accetta un id numerico e fa il fetch di un singolo prodotto da un database usando quell'id. Se il database non ha un prodotto con quell'id in memoria, la fetch fa un throw new Error('Lunghezza 0');
```

## 🔄 3. Navigazione Programmatica (`useNavigate`) vs Dichiarativa (`<Navigate/>`)

React Router offre due approcci radicalmente diversi per spostare l'utente tra le pagine del sito tramite codice.

### 🕹️ `useNavigate()` (Approccio Funzionale/Imperativo)

* **Cos'è:** È una funzione JavaScript pura generata dall'omonimo Hook.


* **Quando usarlo:** Quando la navigazione deve avvenire a seguito di un'azione o di un evento asincrono (es. dopo il successo di un form, al termine di un timer, o al click su un pulsante generico).


* **Nota sulla cronologia:** `navigate(-1)` rimanda indietro l'utente di uno step nella storia del browser. Attenzione: se l'utente è arrivato sulla pagina direttamente da un motore di ricerca esterno, `navigate(-1)` lo rispedirà sul motore di ricerca, non all'interno della tua applicazione.


* **Uso:** `const navigate = useNavigate(); navigate('/dashboard');`


### 🧩 `<Navigate/>` (Approccio Componente/Dichiarativo)

* **Cos'è:** È un componente JSX autosigillante che scatena un reindirizzamento immediato non appena viene renderizzato da React.


* **Quando usarlo:** Ideale per i reindirizzamenti strutturali fissi (es. se l'utente non ha i permessi per vedere una rotta, rimandalo alla home istantaneamente).


* **Pericolo d'uso:** Se inserito nudo e crudo nel corpo principale del JSX senza una condizione di protezione (`if`), renderà la pagina totalmente inaccessibile, rispedendo l'utente altrove all'infinito.

## 🛡️ 4. Autenticazione e Rotte Protette con la Prop `children`

Unendo il controllo dichiarativo di `<Navigate/>` e la flessibilità della prop strutturale `children`, possiamo costruire un guardiano per bloccare l'accesso alle aree riservate del sito (es. Dashboard, Profilo, Carrello).

### 🔒 Il Guardiano: `PrivateRoute.jsx`

```jsx
import { Navigate } from "react-router";

// Il componente riceve lo stato di login e i componenti protetti racchiusi al suo interno
function PrivateRoute({ isLogged = false, children }) {
  if (isLogged === false) {
    // Rimbalzo immediato e dichiarativo alla rotta pubblica radice
    return <Navigate to="/" />;
  } else {
    // Se loggato, dà il via libera e renderizza i componenti interni
    return children;
  }
}

export default PrivateRoute;

```

### 🧱 Applicazione dell'architettura in `App.jsx`

Grazie alla prop `children`, possiamo avvolgere i nostri componenti sensibili direttamente all'interno del guardiano, centralizzando la logica di controllo:

```jsx
<Routes>
  {/* Rotta pubblica accessibile a tutti */}
  <Route path="/" element={<Home />} />

  {/* Rotta protetta: AdminPanel viene passato come "children" a PrivateRoute */}
  <Route 
    path="/admin" 
    element={
      <PrivateRoute isLogged={isUserAuthenticated}>
        <AdminPanel />
      </PrivateRoute>
    } 
  />
</Routes>

```

## 🎨 5. Link Intelligenti con `<NavLink>` e Classi Dinamiche

Il componente `<NavLink>` estende le funzionalità del classico `<Link>`. È progettato specificamente per le barre di navigazione perché è in grado di sapere autonomamente se l'URL del browser corrisponde al proprio attributo `to`.

### 🛠️ Gestione Avanzata con i CSS Modules

Invece di affidarsi alla classe standard `.active` iniettata globalmente nel DOM, le applicazioni moderne estraggono lo stato di attivazione tramite una **funzione di callback applicata all'attributo `className**`, integrando la sicurezza dei CSS Modules per prevenire conflitti di stile.

```jsx
import { NavLink } from 'react-router';
import styles from './Navbar.module.css'; // Importazione del foglio di stile isolato

function Navbar() {
  return (
    <nav>
      <ul>
        <li>
          <NavLink 
            to="/product-list"
            // La callback riceve un oggetto da cui destrutturiamo la proprietà booleana isActive
            className={({ isActive }) => (isActive ? styles.linkAttivo : styles.linkBase)}
          >
            Lista Prodotti
          </NavLink>
        </li>
      </ul>
    </nav>
  );
}

```

### 🧐 Perché questa sintassi contratta è lo Standard di Settore?

1. **Destructuring immediato (`{ isActive }`):** Estrae la proprietà booleana direttamente dai parametri di input della funzione, ripulendo il codice dal rumore visivo (evita di dover scrivere `state.isActive`).


2. **Operatore Ternario Elegante:** La struttura `condizione ? valoreVero : valoreFalso` sostituisce i lunghi costrutti `if/else`, permettendo di capire istantaneamente quale stile dipenda dallo stato attivo.


3. **Return Implicito delle Arrow Functions:** Omettendo le parentesi graffe `{}` e la parola chiave `return`, la logica si sviluppa su un'unica riga leggibile, pulita e facilmente manutenibile nel tempo.
